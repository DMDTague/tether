# Customer observability

Tether should observe whether it delivers connection quickly, reliably, and safely.

## Observe aggressively

- Intent to start, join, knock, invite, Pulse, and queue.
- Join latency and failure codes.
- Provider pairing and match method.
- Synchronization drift and correction.
- Thirty-second and five-minute survival.
- Repeat co-listening and automatic Anchor creation.
- Privacy changes, fast exits, blocks, mutes, and reports.

## Do not observe by default

- Message bodies.
- Verbatim search text.
- Exact coordinates.
- Contact details.
- Authentication or provider tokens.
- Dating answers or safety-report content.

Client events describe intent and perceived errors. Server events remain authoritative for delivery, duration, synchronization, and successful joins.
